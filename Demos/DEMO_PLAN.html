üìö Key Learnings from Demo 3 - Declarative Routing

Backend patterns and architectural decisions for future demos

üó∫Ô∏è The zServer.*.yaml Pattern

  • File Naming: Use zServer_*.yaml (e.g., zServer_routes.yaml) - "zServer" in filename triggers parser detection
  • Parser Flow: zcli.loader.handle() ‚Üí parser_file.py detects "zServer" ‚Üí vafile_server.py parses
  • Structure: Meta (base_path, default_route, error_pages) + routes (path mappings)
  • RBAC: _rbac: {require_role: "roleName"} enforced by HTTPRouter.check_access()
  • Error Pages: error_pages: {403: "access_denied.html"} in Meta ‚Üí auto-served on access denial

üîß Backend Integration Points

  • HTTPRouter: Initialized in zServer._load_routes(), passed to handler via partial()
  • Route Matching: Exact match ‚Üí Wildcard (/*) ‚Üí Default route (from Meta)
  • RBAC Check: z.auth.has_role(role) called in router.check_access() before serving
  • Fallback: If no routes_file, handler.do_GET() uses super() for pure static serving

üé® UX Best Practices

  • Error Feedback: Use auto-redirect countdown (10s) for denied access - clear, user-friendly
  • Animation: Pulsing countdown via CSS @keyframes draws attention without being jarring
  • User Control: Allow cancel via click event - respects user agency
  • Static Assets: Explicitly route CSS/JS files in zServer_routes.yaml to ensure proper serving

üß™ Testing Patterns

  • Mock Files: Place mock zServer.*.yaml in zTestRunner/zMocks/ for test suite
  • RBAC Testing: Mock z.auth.has_role() with return_value=False/True
  • Route Testing: Test exact match, wildcard, default separately for complete coverage
  • Error Pages: Verify error_pages dict keys are int (403), not str ("403")
