Validation

Guide users with instant feedback on their form input using HTML5 validation and zTheme styling

Understanding Form Validation

What is Validation?

Form validation checks if user input meets your requirements before submission. For example:

  • Required fields: "Please fill out this field"
  • Email format: "Please enter a valid email address"
  • Password strength: "Password must be at least 8 characters"

Why Validate?

  • Catch errors early: Users fix mistakes before submitting
  • Save server resources: Invalid data never reaches your backend
  • Improve UX: Instant feedback feels responsive and helpful

Three Validation Approaches

  1. Browser Default: Built-in HTML5 validation (easiest, fully accessible)
  2. Custom Styling: zTheme-styled feedback messages (visually cohesive)
  3. Server-Side: Backend validates and returns feedback (most secure)
Accessibility: Browser default validation is the most accessible option. Custom validation styles may not be announced by screen readers. For production, consider combining browser validation (for accessibility) with server-side validation (for security).

Browser Default Validation

The simplest approach: use HTML5 validation attributes like required, type="email", minlength, and pattern. The browser handles everything:

Letters and numbers only
<form>
  <label for="email" class="zLabel">Email</label>
  <input type="email" class="zInput" id="email" required>
  
  <label for="password" class="zLabel">Password</label>
  <input type="password" class="zInput" id="password" minlength="8" required>
  
  <button type="submit" class="zBtn zBtn-primary">Submit</button>
</form>

Common HTML5 Validation Attributes

  • required - Field must be filled out
  • type="email" - Must be valid email format
  • type="url" - Must be valid URL
  • minlength="8" - Minimum character count
  • maxlength="100" - Maximum character count
  • pattern="[A-Z]+" - Must match regex pattern
  • min="1" / max="10" - For number inputs
Try it: Leave the email field empty or enter invalid data, then click "Create Account". The browser will show its native validation messages (style varies by browser and OS).

How It Works

  1. novalidate: Disables browser default tooltips (we'll style our own)
  2. .zValidation: Tells JavaScript to handle validation
  3. .zWas-validated: Added by JavaScript after submit attempt
  4. .zValid-feedback: Green message shown when input is valid
  5. .zInvalid-feedback: Red message shown when input is invalid
Try it: Click "Submit Form" without filling in Email or City. You'll see red borders and error messages. Fill them in correctly and submit again to see green borders and success messages.

Required JavaScript

To enable custom validation, you have two options:

Option 1: Use zTheme.js (Recommended)

Include the centralized JavaScript file that handles validation automatically:

<script src="/static/js/zTheme.js"></script>

This file auto-initializes all .zValidation forms on page load. No additional code needed!

Option 2: Manual Implementation

If you prefer to implement it yourself or need custom behavior:

<script>
(function() {
    'use strict';
    window.addEventListener('load', function() {
        var forms = document.querySelectorAll('.zValidation');
        Array.prototype.slice.call(forms).forEach(function(form) {
            form.addEventListener('submit', function(event) {
                if (!form.checkValidity()) {
                    event.preventDefault();
                    event.stopPropagation();
                }
                form.classList.add('zWas-validated');
            }, false);
        });
    }, false);
})();
</script>
Note: When using zBifrost client, zTheme.js will be automatically loaded with the theme. No manual inclusion needed.

Server-Side Validation

Server-side validation validates data on the backend after form submission. The server validates the data, then returns HTML with .zValid or .zInvalid classes applied to show results.

How It Works

  1. User submits form ‚Üí Data sent to backend
  2. Backend validates ‚Üí Checks database, business rules, etc.
  3. Server re-renders form ‚Üí Adds .zValid or .zInvalid classes
  4. User sees feedback ‚Üí Green borders for valid, red for invalid

Backend Implementation

Your backend validates the data and renders the form with appropriate classes:

<!-- Backend (Python/Flask example) -->
{% raw %}{% if errors.email %}{% endraw %}
  <input type="email" class="zInput zInvalid" value="{% raw %}{{ email }}{% endraw %}">
  <div class="zInvalid-feedback">{% raw %}{{ errors.email }}{% endraw %}</div>
{% raw %}{% elif email %}{% endraw %}
  <input type="email" class="zInput zValid" value="{% raw %}{{ email }}{% endraw %}">
  <div class="zValid-feedback">Email is available!</div>
{% raw %}{% else %}{% endraw %}
  <input type="email" class="zInput" value="">
{% raw %}{% endif %}{% endraw %}

The key: your server-side code adds .zValid or .zInvalid classes based on validation results.

When to Use Server-Side

  • Database checks: "Is this username/email already registered?"
  • Security validation: "Does this password meet our security policy?"
  • Business logic: "Is this promo code valid and not expired?"
  • Payment processing: "Is this credit card valid?"
  • Always for critical data: Never trust client-side validation alone
Best Practice: Use client-side validation for instant UX feedback, but always validate on the server for security. Client-side validation can be bypassed by malicious users.

Best Practices

Write Helpful Messages

  • ‚ùå "Invalid input"
  • ‚úÖ "Email must include an @ symbol"
  • ‚ùå "Error"
  • ‚úÖ "Password must be at least 8 characters"

Validate on Submit, Not on Type

Don't show errors while the user is still typing. Wait until they attempt to submit (or blur the field). This avoids frustrating "Email is invalid" messages before they've finished typing.

Always Validate Server-Side

Client-side validation improves UX but can be bypassed. Always validate on the server for security and data integrity.

Use Native Validation When Possible

Browser default validation is accessible, tested, and works without JavaScript. Consider it for simpler forms.

Test with Screen Readers

Custom validation may not announce errors to assistive technologies. Test with screen readers or stick to browser defaults for critical forms.
