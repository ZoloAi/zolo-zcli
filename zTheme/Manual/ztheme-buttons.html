Buttons

Use zTheme's button styles for actions in forms, dialogs, and more with support for multiple sizes, states, and color variants

Understanding Buttons

What are Buttons?

Buttons are interactive elements that trigger actions when clicked. They're one of the most important UI components because they:

  • Signal actions: Submit forms, save data, navigate pages
  • Guide users: Clear calls-to-action for user journeys
  • Provide feedback: Visual states (hover, active, disabled) communicate interactivity

When to Use Buttons

  • Form submissions: Use the <button type="submit"> HTML element for submitting data
  • In-page actions: Use the <button> HTML element to open modals, toggle content, trigger JavaScript
  • Navigation: Use the <a href="/page"> HTML element for links, then add zTheme button classes like class="zBtn zBtn-primary" to style it as a button
Semantic HTML vs Visual Styling: The HTML element defines the meaning (<button> for actions, <a> for navigation), while zTheme utility classes like .zBtn and .zBtn-primary define the visual appearance. This separation helps screen readers and search engines understand your page correctly. When styling an <a> element as a button, also add role="button" for accessibility.

The Button Element

HTML provides the semantic <button> element for clickable actions. Our reboot provides basic, clean styling for all buttons:

<button>Default Button</button>
<button type="submit">Submit Button</button>
<button type="reset">Reset Button</button>

What the reboot provides:

  • Clean, consistent cross-browser styling
  • Proper padding and font inheritance
  • Cursor pointer on hover
  • Focus states for accessibility
  • Disabled state opacity

Enhanced Buttons with .zBtn

The .zBtn class adds zTheme's enhanced styling on top of the semantic <button>. Combined with color variants, you get professional, accessible buttons:

<button class="zBtn zBtn-primary">Primary</button>
<button class="zBtn zBtn-secondary">Secondary</button>
<button class="zBtn zBtn-success">Success</button>
<button class="zBtn zBtn-info">Info</button>
<button class="zBtn zBtn-warning">Warning</button>
<button class="zBtn zBtn-danger">Danger</button>
<button class="zBtn zBtn-light">Light</button>
<button class="zBtn zBtn-dark">Dark</button>
Accessibility Note: Button colors should convey meaning (e.g., red for danger, green for success), but also include clear text labels. Don't rely on color alone—users with color blindness need text context.

Outline Buttons

For a lighter visual weight, use outline variants. These buttons have transparent backgrounds with colored borders and text:

<button class="zBtn zBtn-outline-primary">Primary</button>
<button class="zBtn zBtn-outline-secondary">Secondary</button>
<button class="zBtn zBtn-outline-success">Success</button>
<button class="zBtn zBtn-outline-info">Info</button>
<button class="zBtn zBtn-outline-warning">Warning</button>
<button class="zBtn zBtn-outline-danger">Danger</button>
<button class="zBtn zBtn-outline-light">Light</button>
<button class="zBtn zBtn-outline-dark">Dark</button>
Contrast Note: Some outline button styles use a relatively light foreground color, and should only be used on a dark background for sufficient contrast. Specifically, .zBtn-outline-light is designed for dark backgrounds.

When to use outline buttons:

  • Secondary actions alongside a primary button
  • Lighter visual hierarchy (less visual "weight")
  • Better contrast on colored backgrounds

Button Sizes

Add .zBtn-lg or .zBtn-sm to adjust button size for different contexts:

<button class="zBtn zBtn-primary zBtn-lg">Large Button</button>
<button class="zBtn zBtn-primary">Default Button</button>
<button class="zBtn zBtn-primary zBtn-sm">Small Button</button>

Size guidelines:

  • Large (.zBtn-lg): Hero sections, primary CTAs
  • Default: Most buttons (forms, actions)
  • Small (.zBtn-sm): Tight spaces, tables, inline actions

Disabled State

Use the disabled attribute to prevent interaction and reduce opacity. This visually communicates that an action is unavailable:

Disabled Link 
<!-- Button element: use disabled attribute -->
<button class="zBtn zBtn-primary" disabled>Disabled Primary</button>

<!-- Link element: use .zDisabled class + aria-disabled -->
<a href="#" class="zBtn zBtn-info zDisabled" tabindex="-1" 
   role="button" aria-disabled="true">Disabled Link</a>
Links vs Buttons: The disabled attribute doesn't work on <a> elements. For links styled as buttons, add .zDisabled, tabindex="-1", and aria-disabled="true" for accessibility.

Block Buttons

Add .zBtn-block to make a button span the full width of its container. Perfect for mobile layouts and form submissions:

<button class="zBtn zBtn-primary zBtn-block">Full Width Button</button>
<button class="zBtn zBtn-outline-secondary zBtn-block">Another Block Button</button>

Responsive block buttons:

  • .zBtn-block-sm - Full width on screens ≤ 576px
  • .zBtn-block-md - Full width on screens ≤ 768px

Link Buttons

The .zBtn-link variant styles a button to look like a text link. Use for tertiary actions that need less visual weight:

<button class="zBtn zBtn-link">Link Button</button>
<button class="zBtn zBtn-link" disabled>Disabled Link</button>

Using Links as Buttons

You can style <a> elements as buttons for navigation. Remember to add role="button" for accessibility:

Link to zTheme Forms Overview 
<a href="ztheme.html" class="zBtn zBtn-primary" role="button">Link to zTheme</a>
<a href="ztheme-forms-overview.html" class="zBtn zBtn-outline-info" role="button">Forms Overview</a>
Semantic Rule: Use <button> for actions (submit, toggle) and <a> for navigation. This helps screen readers and search engines understand your page structure.

Best Practices

  • Use semantic HTML: <button> for actions, <a> for links
  • Clear labels: Button text should describe the action ("Save Changes" not just "OK")
  • Color meaning: Red for destructive actions (delete), green for positive (save, submit)
  • Visual hierarchy: One primary button per section, use outline/link for secondary actions
  • Disabled feedback: Use disabled attribute, not just visual styling
  • Mobile-friendly: Ensure buttons have at least 44×44px touch target for mobile
  • Keyboard accessible: All buttons should be reachable via Tab key
