Understanding Tooltips

Tooltips are small contextual pop-ups that appear when users hover over or focus on an element. They provide brief, helpful information without requiring a click or cluttering the interface.

Perfect for:

  • Icon explanations - "Save document"
  • Abbreviated text - Full form of acronyms
  • Feature hints - Brief descriptions of buttons
  • Additional context - Extra info without visual clutter
  • Link previews - Hint at destination

‚ö†Ô∏è Tooltips vs Popovers:

  • Tooltips: Hover-triggered, plain text only, simple hints
  • Popovers: Click-triggered, rich HTML content, more complex info

Use tooltips for brief hints, popovers for detailed explanations.

üì± Mobile Limitation:

Tooltips don't work on touch devices! Mobile phones and tablets have no hover state. On touch screens, users must tap and hold to see tooltips, which is not intuitive. Never put critical information only in tooltips. Always ensure essential content is visible or accessible through other means (labels, help text, etc.). Consider using the focus trigger for better mobile support, or use click-triggered popovers instead.

Trigger Options

Control when tooltips appear with the data-bs-trigger attribute:

<button data-bs-toggle="tooltip" data-bs-trigger="hover" title="...">Hover</button>
<button data-bs-toggle="tooltip" data-bs-trigger="focus" title="...">Focus</button>
<button data-bs-toggle="tooltip" data-bs-trigger="click" title="...">Click</button>

Disabled Elements

Disabled elements can't receive focus or hover events. Wrap them in a <span> to enable tooltips:

<span data-bs-toggle="tooltip" title="This button is disabled">
  <button class="zBtn zBtn-primary" disabled>
    Disabled Button
  </button>
</span>

Best Practices

‚úÖ DO:

  • Keep tooltip text brief (1-2 short sentences max)
  • Use tooltips for icon-only buttons
  • Provide keyboard access (hover + focus triggers)
  • Use consistent placement across similar elements
  • Explain abbreviations and acronyms
  • Test with screen readers
  • Use plain, helpful language

‚ùå DON'T:

  • Put critical information only in tooltips - Mobile/tablet users can't hover, so they'll miss it entirely!
  • Use long paragraphs (use popovers instead)
  • Add tooltips to every element (causes tooltip fatigue)
  • Nest interactive elements inside tooltips
  • Design for desktop only - Over 50% of web traffic is mobile. Test without hover!
  • Use tooltips on disabled elements without wrapper
  • Rely solely on hover (always include focus trigger for keyboard users)

üì± Mobile-Friendly Alternatives:

  • Use visible labels or help text instead of tooltips
  • Use data-bs-trigger="focus" so tapping shows the tooltip
  • Consider click-triggered popovers for important information
  • Add an info icon (‚ÑπÔ∏è) that users can tap for help
  • Show tooltips content in the UI on small screens with CSS @media queries

Accessibility

Tooltips must be accessible to keyboard and screen reader users:

Keyboard Navigation

  • Tab: Move focus to tooltip trigger (shows tooltip)
  • Shift+Tab: Move focus away (hides tooltip)
  • Esc: Close tooltip (if trigger="click")

Screen Reader Support

  • ARIA Role: Tooltips automatically get role="tooltip"
  • Content: Tooltip text is announced when element receives focus
  • Dismissal: Announced when tooltip disappears

Important Considerations

  • Mobile/Touch Devices: Tooltips require hover, which doesn't exist on phones/tablets. Users must long-press to see them, which is unintuitive. This is the #1 accessibility issue with tooltips!
  • Critical Info: Never put essential information only in tooltips. Always provide alternative access (visible labels, help text, expandable sections).
  • Focus Indicators: Ensure focusable elements have visible focus styles
  • Reduced Motion: Fade animations respect prefers-reduced-motion
  • Testing: Always test your interface with tooltips disabled or on a touch device to ensure usability
