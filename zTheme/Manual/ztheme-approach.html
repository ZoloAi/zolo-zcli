Approach

The guiding principles behind zTheme's design and implementation

Learn about the guiding principles, strategies, and techniques used to build and maintain zTheme. Understanding these principles will help you use zTheme more effectively and extend it for your projects.

Summary

At a high level, here's what guides zTheme's approach:

  • Components are responsive and mobile-first - Design for small screens, enhance for larger
  • Base class + modifiers pattern - Common styles in base, variations in modifiers
  • Z-index scale - Predictable layering across all components
  • Prefer HTML/CSS over JavaScript - Simpler, faster, more accessible
  • Utilities over custom styles - Reuse classes, reduce CSS bloat
  • Flexible HTML requirements - Avoid strict parent-child selectors
  • Z prefix convention - All utility classes use .z prefix for clarity

Responsive & Mobile-First

zTheme is built with a mobile-first approach. This means styles are written for small screens first, then enhanced for larger viewports using min-width media queries.

Why Mobile-First?

  • Reduces CSS overrides - Add styles as viewport grows instead of removing them
  • Better performance - Mobile devices load only what they need
  • Forces prioritization - Design for constraints first, enhance later

Example

/* Mobile-first: starts with mobile, enhances for desktop */
.zContainer {
  width: 100%;              /* Mobile default */
  padding: 0 1rem;
}

@media (min-width: 768px) { /* Tablet and up */
  .zContainer {
    max-width: 720px;
  }
}

@media (min-width: 1200px) { /* Desktop */
  .zContainer {
    max-width: 1140px;
  }
}

Base Class + Modifiers Pattern

All zTheme components follow a consistent pattern: a base class defines common properties, and modifier classes add variations.

Example: Buttons

/* Base class - common properties */
.zBtn {
  display: inline-block;
  padding: 0.5rem 1rem;
  border: 1px solid transparent;
  border-radius: 4px;
  font-size: 1rem;
  cursor: pointer;
}

/* Modifiers - variations */
.zBtn-primary { background: #007bff; color: white; }
.zBtn-success { background: #28a745; color: white; }
.zBtn-lg { padding: 0.75rem 1.5rem; font-size: 1.25rem; }

üí° Why this pattern? It prevents code duplication and makes overrides predictable. Change the base class to affect all buttons, or target specific modifiers for variations.

Z-Index Scale

zTheme uses two predictable z-index scales to manage layering:

Component Elements (0-3)

Value State Purpose
0 Default Initial state
1 :hover User intent
2 :active / .active Active state
3 :focus User attention (highest priority)

Overlay Components (1000+)

Component Typical z-index
Dropdowns 1000
Fixed/Sticky Navbar 1020
Tooltips & Popovers 1070
Modals 1050
Offcanvas 1045

HTML & CSS Over JavaScript

Whenever possible, zTheme prefers HTML and CSS solutions over JavaScript:

Why?

  • Performance - CSS is faster than JavaScript for styling
  • Accessibility - HTML semantics work with assistive technologies
  • Simplicity - Easier to understand and maintain
  • Progressive enhancement - Works even if JavaScript fails

Example: Hover Effects

‚ùå Avoid (JavaScript)

element.addEventListener(
  'mouseenter', 
  () => {
    element.style.background 
      = '#007bff';
  }
);

‚úÖ Prefer (CSS)

.zBtn:hover {
  background: #007bff;
}

üí° When to use JavaScript: For complex interactions like modals, dropdowns with keyboard navigation, or dynamic content loading. JavaScript enhances, it doesn't replace.

Utilities Over Custom Styles

Utility classes are a powerful tool for building interfaces quickly while keeping CSS lean:

Benefits

  • Speed - Build layouts faster without writing custom CSS
  • Consistency - Standardized spacing, colors, and sizing
  • Performance - Reuse classes instead of duplicating styles
  • Maintainability - Changes in one place affect all usage

Example Comparison

Custom CSS

/* Custom styles */
.card-header {
  margin-bottom: 2rem;
  padding: 1.5rem;
  background: #f8f9fa;
  border-radius: 4px;
}

/* Repeated everywhere */

Utility Classes

<div class="zmb-4 zp-3 
            zBg-light 
            zRounded">
  Content
</div>

/* Reusable everywhere */

‚ö†Ô∏è Balance: Use utilities for common patterns, create component classes for complex, repeated structures. Don't over-utility every single element.

Z Prefix Convention

All zTheme utility classes use the .z prefix for clarity and namespace protection:

Why Z Prefix?

  • Namespace protection - Prevents conflicts with other CSS frameworks
  • Visual clarity - Instantly recognizable as zTheme utilities
  • zCLI ecosystem - Consistent with z-prefixed subsystems (zConfig, zComm, zServer, etc.)
  • Easy to search - Find all zTheme classes by searching for "z"

Examples

/* Layout */
.zContainer, .zRow, .zCol-*

/* Spacing */
.zm-*, .zp-*, .zGap-*

/* Typography */
.zText-*, .zFs-*, .zFw-*

/* Display */
.zFlex, .zD-block, .zAlign-*

/* Colors */
.zText-primary, .zBg-success, .zBorder-danger

Flexible HTML

zTheme avoids strict HTML requirements when possible. We focus on single classes and avoid immediate child selectors (>).

Why?

  • Flexibility - Use components in different contexts
  • Simpler CSS - Lower specificity means easier overrides
  • Better maintainability - Changes don't cascade unexpectedly

Example

‚ùå Too Strict

/* Requires exact structure */
.card > .card-body > h3 {
  font-size: 1.5rem;
}

/* Hard to override */

‚úÖ Flexible

/* Works anywhere */
.card-title {
  font-size: 1.5rem;
}

/* Easy to override */

Best Practices Summary

‚úÖ Do

  • Design mobile-first, enhance for larger screens
  • Use base class + modifier pattern for components
  • Leverage utility classes for common patterns
  • Follow the established z-index scales
  • Prefer CSS over JavaScript for styling
  • Use semantic HTML elements when possible
  • Keep specificity low for easier overrides

‚ùå Don't

  • Write desktop-first styles that need mobile overrides
  • Create custom z-index values without reason
  • Over-utility every element (balance is key)
  • Use JavaScript for what CSS can do
  • Write overly specific selectors
  • Ignore semantic HTML in favor of divs
